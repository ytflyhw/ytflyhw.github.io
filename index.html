MySQL多版本并发控制（MVCC）

版本链

在InnoDB引擎中，它的聚簇索引记录有两个必要的隐藏列

  • trx_id

    这个id用来存储每次对这条聚簇索引记录进行修改的事务id

  • roll_pointer

    每次对哪条聚簇索引记录有修改的时候，都会把老版本写入undo日志。这个roll_pointer就是存了一个指针指向这条聚簇索引记录的上一个版本位置，通过它获得上一个版本信息。

ReadView

“已提交读”与“可重复读”的区别就在于他们的ReadView策略不同。

ReadView中主要就是有个列表来存储我们系统中当前活跃着的读写事务，也就是begin了还未提交的事务。通过这个列表来判断记录的某个版本是否对当前事务可见。

  • 已提交读

    事务在每次查询的开始都会生成一个独立的ReadView

  • 可重复读

    事务在第一次读的时候生成一个ReadView，之后的读都复用之前的ReadView

2.2 更多信息

使用-x参数我们可以获得更多统计信息。

rrqm/s：每秒这个设备相关的读取请求有多少被Merge了（当系统调用需要读取数据的时候，VFS将请求发到各个FS，如果FS发现不同的读取请求读取的是相同Block的数据，FS会将这个请求合并Merge）；

wrqm/s：每秒这个 设备相关的写入请求有多少被Merge了。

wsec/： 每秒写入的扇区数。

r/s：每秒读请求数；

w/s：每秒写请求数；

await：每一个IO请求的处理的平均时间（单位是微秒）。这里可以理解为IO的响应时间，一般地系统IO响应时间应该低于5ms，如果大于10ms就比较大了。

%util：在统计时间内所有处理IO时间，除以总共统计时间。例如，如果统计间隔1秒，该 设备有0.8秒在处理IO，而0.2秒闲置，那么该设备的%util = 0.8/1 = 80%，所以该参数暗示了设备的繁忙程度。一般地，如果该参数是100%表示设备已经接近满负荷运行了（当然如果是多磁盘，即使%util是100%，因 为磁盘的并发能力，所以磁盘使用未必就到了瓶颈）。

Java垃圾处理

1、垃圾回收算法

1.1、标记-清除

标记所有需要回收的对象，标记完成后统一回收。

不足：

  • 原地清除会产生大量的不连续的内存碎片，碎片太多可能导致需要分配大对象时找不到足够的连续内存，而不得不提前触发另一个垃圾回收动作。

1.2、标记-复制

将可用内存划分为大小相等的两块，每次只使用其中的一块，当这块用完了就将还存活的对象复制到另一块内存上，将使用过的内存清理掉。

不足：

  • 可用的内存空间缩小为原来的一半，为了应对被使用的内存中的所有对象都100%存活的极端情况还需要额外的空间进行担保。

1.3、标记-整理

标记出需要回收的对象后不是直接回收，而是让所有存活的对象都向一端移动，然后直接清理掉边界之外的内存。

不足：

  • 需要移动对象的内存位置，效率不高。

2、标记算法

如何确定那些对象不再被引用成为了垃圾呢，这就需要根搜索法

  • 首先需要枚举GC root节点，之后以这些节点为根去分析对象的可达性，即是否还有引用指向某个对象
  • 可达性的分析必须在一个能确保一致性的快照中进行，即分析的过程中整个系统是冻结的不再产生新的引用变化，这就需要阻塞全部的用户线程（Stop the world）

2.1、三色标记法

在并发标记时，系统是动态的，我们需要一种方法推演回收器的正确性

  • 黑色:根对象，或者该对象与它的子对象都被扫描

  • 灰色:对象本身被扫描,但还没扫描完该对象中的子对象

  • 白色:未被扫描对象，扫描完成所有对象之后，最终为白色的为不可达对象，即垃圾对象

3、分代收集算法

将Java堆分为新生代和老年代。

  • 新生代中每次回收垃圾都会有大批的对象死去，只有少量存活，可以使用“标记-复制”算法，每次只需要复制少量存活的对象即可完成回收。
  • 老年代中对象存活率高，且再没有额外空间对其进行担保，需要“标记-清理”或“标记整理”。

有研究表明98%的对象都是“朝生夕死”，所以新生代中也不需要按照1:1去分配内存，而是将内存分为一块较大的“Eden(伊甸园)**”和两块较小的“Survivor(幸存者)**”

对象被创建后首先被放在Eden中，进行垃圾回收后不能被回收的对象会被放入空的Survivor区域。Eden发生垃圾回收时，其中一块Survivor也会执行垃圾回收，将存活对象放入空的Survivor，然后本身被清理，从而保证永远有一块“Survivor”是空的。

4、垃圾回收器

4.1、CMS收集器（并发收集）

CMS(Concurrent Mark Sweep) 收集器是一种以获取最短回收停顿时间为目标的收集器。CMS基于 标记-清除算法实现。整个过程分为4个步骤：

  • 初始标记(CMS initial mark) ，仅仅标记以下GC Roots能直接关联的对象，速度很快。STW

  • 并发标记(CMS concurrent mark)，进行GC Roots Tracing的过程

  • 重新标记(CMS remark) ，修正并发标记期间因为用户程序继续运作而导致标记产生变动的那一部分对象的标记记录。STW

  • 并发清除(CMS concurrent sweep)

不足：

  • CMS收集器对CPU资源非常敏感，默认的回收线程将消耗25%的CPU资源
  • 无法处理浮动垃圾（并发标记时系统产生的垃圾）
  • 基于”标记-清除“算法，会产生大量垃圾碎片。

4.2、G1收集器

  • G1的设计原则就是简单可行的性能调优，程序员只需要声明使用G1，并设置堆内存的最大内存以及GC的最大暂停时间。（MaxGCPauseMillis设置的过小，那么GC就会频繁，吞吐量就会下降。如果MaxGCPauseMillis设置的过大，应用程序暂停时间就会变长。）

  • G1将新生代，老年代的物理空间划分取消了。将堆划分为若干Region，这些区域部分属于新生代，部分属于年老代，新生代垃圾回收时依然将存活对象复制到Survivor或者年老代中（其实就是从这个区域复制到另一个区域），更细致的分区使得G1完成了堆压缩，从而解决了CMS的内存碎片问题。

  • G1中有个特殊区域Humongous用来存放巨大对象（占用的空间超过了分区容量50%）。如果一个H区装不下一个巨型对象，那么G1会寻找连续的H分区来存储（可能触发Full GC）。

  • G1提供了两种收集模式：

    • Young GC，选定所有新生代里的region。通过控制新生代的region个数来控制young GC的开销。
    • Mixed GC，选定所有新生代里的region，外加根据全局并发标记统计得出收集收益高的若干老年代region。在用户指定的开销目标范围内尽可能选择收益高的老年代region。

应用层协议（JCE & ProtoBuf）

两个节点间为了实现信息交换，需要约定一些数据格式的规则，如：字节序，字段类型等。

1、JCE格式编码

1.1、结构体定义

  • 第一列数字为改字段的唯一标识符（Tag），无论增减字段改值必须与相应字段对应（一般不减字段，容易造成Tag被无意复用），Tag取值范围为【0，255】；
  • 第二列表示该字段是否必选；
    • require 表示该字段必选，缺失该 Tag 解码将失败；
    • optional 表示该字段可选，确实该 Tag 则忽略。
  • 第三列为该字段的数据类型，这决定之后有多少个字节用于存储该字段的具体值。
  • key 表示结构的比较操作，缺省 key 时该 struct 不可比较，key[Stuct, member…]：
    • Struct 表示结构名称
    • Member 表示成员变量，比较时按照顺序进行优先比较。

1.2、基本结构

JCE使用TTLV编码（ <Tag，Type，Length，Value>四元组编码 ），编码由头信息和实际信息组成。

其中头信息包含：

Type Tag1 Tag2
4 bits 4bits 1byte

1.2.1、Type由4位二进制表示，取值范围0~15

取值 类型 备注
0 int1 紧跟1个字节整型数据
1 int2 紧跟2个字节整型数据
2 int4 紧跟4个字节整型数据
3 int8 紧跟8个字节整型数据
4 float 紧跟4个字节浮点型数据
5 double 紧跟8个字节浮点型数据
6 String1 紧跟1个字节长度，再跟内容
7 String4 紧跟4个字节长度，再跟内容
8 Map 紧跟一个整型数据表示Map的大小，再跟[key, value]对列表
9 List 紧跟一个整型数据表示List的大小，再跟元素列表
10 自定义结构开始 自定义结构开始标志
11 自定义结构结束 自定义结构结束标志，Tag为0
12 数字0 表示数字0，后面不跟数据
13 SimpleList 简单列表（目前用在byte数组），紧跟一个类型字段（目前只支持byte），紧跟一个整型数据表示长度，再跟byte数据
14 - -
15 - -

1.2.2、Tag 由 Tag1 和 Tag2一起表示，取值范围是0~255

Tag 用来区分不同的字段（编码之后的数据中，只会存储字段对应的tag，而不会传输字段名）。其中 Tag2 是可选的，当 Tag 的值不超过14时，只需要用 Tag1 就可以表示；当 Tag 的值超过14而小于256时，Tag1 固定为15，而用 Tag2 表示 Tag 的值。

2、ProtoBuf（PB）格式编码

2.1、结构体定义

等号右边数字为唯一标识符（Tag），其他元素含义与 JCE 编码向类似

2.2、Base 128 Varints

PB格式编码采用 Base 128 Varints 存储方式：每块数据由接连的若干个字节表示（小的数据用1个字节就可以表示），每个字节最高位标识本块数据是否结束（1:未结束，0:结束），低7位表示数据内容。（可以看出数据封包后体积至少增大14.2%）

字节序为小端字节序，即数据的第一个字节在最后。即 300 = 1010 1100 0000 0010

2.2.1 ZigZag 编码解决符号数问题

普通的补码导致绝对值较小的负数（恰恰是常用的那些）在传输时不得不传输大量的 1，且 Base 128 Varints 的策略将保留这些 1。这时 ZigZag 编码便能很好的解决问题。

ZigZag 编码先将负数取补，再将符号位移到数据的最末端，最后将将剩余位取反，以 2 byte 为例：

  • -1 = [11111111_11111111]补= [11111111_11111111]符号后移=[00000000_00000001]其他位取反
  • 1 = [00000000_00000010]

再用 Base 128 Varints 策略即可很好的避免传输容易数据。

2.3、ProtoBuf的优缺点

优点：

  • 序列化和反序列化效率比 xml 和 json 都高
  • 字段可以乱序，欠缺，因此可以用来兼容旧的协议，或者是减少协议数据。（当然 JCE 也支持）

缺点（这两个确定对于没有指针的语言来说，代价将相当昂贵）：

  • 如果字段过多，或者嵌套过深，都会影响反序列化效率，解析每一块数据都要根据序号找到对应的位置然后再插入到已解析好的数据中。

1、基本原则

  • GoLang 是一门编译型语言，程序员写的代码将会被直接编译为可执行文件（没有Python一类的解释器）

  • 开发需要配置GOPATH，这是之后开发的根目录

  • 所有程序一定属于某个package，需要在程序的最初进行声明。main包下的main函数将是整个程序的入口。

  • 所有 import 了的包，函数内声明的变量都必须被使用，否则编译将会报错。

  • 不必在意缩进，空格等对齐问题，内置的 go fmt 将会为我们处理好一切

  • 函数外只能写标识符的声明，不能写任何语句。标识符只能以字母或下划线开始。

  • GoLang 所有的参数传递默认都是值传递，如果希望处理变量本身，需要指明参数类型为指针，但函数内不需要可以强调指针。

  • GoLang通过起多个goroutine实现多线程，不同线程之间不同于 JAVA 的共享内存对数据加锁的方式实现同步，而是通过 Channel（类似消息队列，一个线程往里写，一个线程往外读） 实现共享内存。
