再没有IOC的程序中，对象的创建与对象间的依赖关系完全硬编码在程序中，对象的创建由程序自己控制，引入IOC后，对象的创建转移给了第三方，即获得依赖对象的方式反转了。
IOC是Spring框架的核心内容，可以使用XML配置，也可以使用注解，甚至新版本Spring可以零配置实现IOC。IOC容器在初始化时先读取配置文件，根据配置文件或元数据创建与组织对象存入容器，程序使用时再从IOC容器中取出需要的对象。

控制反转是一种描述（XML或注解）并通过第三方去生产或获取特定对象的方式。在Spring中实现控制反转的是IOC容器，其实现方法是依赖注入（Dependency Injection,DI）。

5、Bean的自动装配

自动装配是Spring满足bean依赖的一种方式，Spring会在上下文中自动寻找，自动给bean装配属性。

byName，自动在容器上下文中查找，和自己set方法后面的值对象对应的 bean id（需要保证bean的id唯一，且与set方法的值一致）

byType，自动在容器上下文中查找，和自己对象属性类型相同的 bean id（需要保证所有bean的class唯一）

基于注解自动装配

绝大多数时间均采用注解开发而非xml。

使用注解需要：

  • 导入约束：context约束
  • 配置注解的支持：context:annotation-config/

在需要注入的属性上加入@Autowired注解，Soring将按照名称自动导入（也可以在set方法上使用）

加入@Autowired的属性可以不用编写set方法，前提是这个自动装配的属性在IOC容器中存在，且符合byName原则。

@Autowired(required=false)表示这个属性可以为null否则不可为空。

@Autowired可以与@Qualifier(value=”别名”)配合实现在bean id与属性名不同时的装配。

@Autowired优先按照byType查找，类型重复则按照byName查找。

Java原生注解@Resource也可以实现自动装配，优先按照byName查找，查找不到会再按照byType查找。可以通过name指定”别名“。

8、代理模式

SpringAOP的底层即为代理模式！

代理模式的分类：

  • 静态代理
  • 动态代理

静态代理

角色分析：

抽象角色：一般会使用接口或抽象类解决

真实角色：被代理的角色

代理角色：代理真实角色，代理真实角色后，我们一般会做一些附属操作

客户端访问角色

代理模式的好处：

  • 可以使真实角色的操作更加纯粹，不需要关注一些公共业务！
  • 公共业务交给代理角色，实现业务分工！
  • 公共业务发生扩展时，方便集中管理！

代理模式的缺点：

  • 一个真实角色就会产生一个代理角色；代码量翻倍，开发效率降低。

动态代理

动态代理和静态代理的角色是一样的，可以分为两类：基于接口的动态代理，基于类的动态代理。

动态代理的代理类是动态生成的，不是我们直接写好的，主要用到java.lang.reflect包中的两个类：Proxy以及InvocationHandler.

  • Proxy提供了创建动态代理类和实例的静态方法。
  • InvocationHandler是代理实例的调用处理程序实现的接口，每个代理实例都有的一个关联的调用处理程序。

抽象角色（接口）

真实角色（被代理的角色）

代理处理程序（通用，target可以换成其他需要代理的接口）

客户端访问角色

动态代理保留了静态代理的优势，且一个动态代理类代理的是一个接口，可以代理多个类即代理了一类业务。

9、AOP

AOP（面向切面编程），通过预编译方式和运行期动态代理实现程序功能的统一维护的一种技术

AOP在Spring的作用

提供声明式事务；允许用户自定义切面。

  • 横切关注点：跨越应用程序多个模块的方法或功能，即与业务逻辑无关，但依然需要被关注的部分（例如，日志，安全，缓存，事务等）。
  • 切面（Aspect）：横切关注点被模块化的特殊对象，就是一个类。
  • 通知（Advice）：切面必须完成的工作，即类中的一个方法。
  • 目标（Target）：被通知的对象。
  • 代理（Proxy）：向目标对象应用通知后创建的对象。
  • 切入点（PointCut）：切面通知执行的地点。
  • 连接点（JointPoint）：与切入点匹配的执行点。

SpringAOP支持5中类型的通知：

  • 前置通知：方法前执行（org.springframework.aop.MethodBeforeAdvice）
  • 后置通知：方法后执行（org.springframework.aop.MethodReturningAdvice）
  • 环绕通知：方法前后执行（org.aopalliance.intercept.MethodInterceptor）
  • 异常抛出通知：方法抛出异常时执行（org.springframework.aop.ThrowsAdvice）
  • 引介通知：类中增加新的方法属性（org.springframework.aop.IntroductionIntercepto）

lombok注解使用

  创建一个类往往需要编写这个类的各种构造方法，各个属性的set/get方法，以及重写Object中的方法；而lombok可以让这一切变得简单。只需要再pom文件中引入相关依赖，即可使用一行注解替换大段代码。

@Getter/@Setter

  作用于成员变量，生成该变量的get/set方法；
  作用于类，生成该类所有成员变量的get/set方法。

@NonNull

  作用于成员变量，标识该变量不可为空，否则抛出空指针异常；
  作用于成员方法的参数，标识调用该方法时，该参数不可为空，否则抛出空指针异常。

@NoArgsConstructor/@AllArgsConstructor

  作用于类，生成该类的无参构造/全参构造。

@RequiredArgsConstructor

  作用于类，生成所有被fianl或@NonNull修饰的成员变量的构造方法。

@ToString

  作用于类，重写toString()方法，可以通过of指定要使用的成员变量，也可以通过exclude指定不使用的成员变量。
  例如：@ToString(of={“param1”,”param2”})或@ToString(exclude={“param1”,”param2”})

@EqualsAndHashCode

  作用于类，重写equals()和hashcode(),可以通过of指定要使用的成员变量，也可以通过exclude指定不使用的成员变量。

@Data

  作用于类，是@Getter，@Setter，@ToString，@EqualsAndHashCode，@RequiredArgsConstructor的集合。

@Value

  作用于类，类似于@Data，但默认所有成员变量都由”private final”修饰，因而不会生成set方法。

@Cleanup

  作用于资源变量，标识该资源会被自动关闭，默认调用close()方法，也可以通过@Cleanup(“methodName”)指定要调用的方法。
  例如： @Cleanup InputStream in = new FileInputStream();

@SneakyThrows

  作用于方法，相当于try-catch语句包，捕获异常并在catch中用Lombok.sneakyThrow(e)把异常抛出，可以通过@SneakyThrows(Exception.class)指定抛出的异常类型。

JMeter压力测试

  JMeter是Apache基于java开发的压力测试工具，可以对网络服务模拟巨大负载，无需安装即可在装有JVM的计算机上运行。

线程组

  Test Plan -> 添加 -> 线程（用户）-> 线程组
  在线程组中可以配置线程数， Ramp-Up时间， 循环次数等
  - 线程数，即同时对服务器端口访问的线程数量。
  - Ramp-Up时间，即上述线程在多长时间内完成访问，0即瞬时完成。
  - 循环次数，即上述访问的循环次数。
 

取样器（HTTP请求）

  线程组 -> 添加 -> 取样器 -> HTTP请求
  在取样器中需要配置协议， 服务器地址， 端口， 请求类型， 路径，对于含参请求还需要配置随请求一起发出的参数。
 

配置元件

HTTP请求默认值

  线程组 -> 添加 -> 配置元件 -> HTTP请求默认值
  对于同一服务需要测试多个页面吞吐时，重复配置服务器参数便十分繁琐，可以将其统一配置在HTTP请求默认值元件中。
 

CSV数据文件设置

  线程组 -> 添加 -> 配置元件 -> CSV数据文件设置
  当请求含有复杂参数时，我们可以将其参数保存在单独的txt文件中，再通过CSV数据文件设置加以载入。
 

HTTP Cookie管理器

  对于需要Cookie的请求，便需要HTTP Cookie管理器进行配置。
  注意设置中的”名称”需要与Cookie名称相同，而值需要与CSV数据文件设置中设定的变量名称相同。
 

监听器

  线程组 -> 添加 -> 监听器 -> [聚合报告]
  监听器可以将测试结果加以整理展示。

RabbitMq 笔记

  消息队列是分布式系统中重要的组件，其可以比作是一个存放消息的容器，一边连接着生产者，另一边连接着消费者。生产者负责生产消息，消费者则监听着队列，负责消费（处理）消息。使用消息队列可以实现异步处理，进而实现削峰，提高系统性能，降低系统耦合对。
RabbitMq是由Erlang语言编写的，实现了AMQP协议的开源消息代理中间件，具有可伸缩（集群服务）以及消息持久化（硬盘加载）的特性。

交换模式

  根据消息队列所使用的交换机（Exchange）类型不同，可以将消息交换模式分为：direct, topic, header, fanout。其中header交换机极少使用，本文不做介绍。

fanout

  fanout交换机是最简单的交换机，它不会处理RoutingKey，直接将生产者产生的消息’广播’给所有的队列，最终由监听这些队列的消费者将消息消费掉。
 

direct

  不同于fanout的广播模式，direct交换机将根据每一条消息的RoutingKey决定他们应当进入的队列。
 

topic

  direct交换机的确为我们处理消息的去向提供了便利，但当队列增多时，RoutingKey自然也会增多，管理这些RoutingKey也会越来越困难。topic交换机通过引入通配符的方式解决了这一问题。
  我们以 . 将RoutingKey划分为数个单词，
  * 代表一个单词。
  # 代表0个或多个单词。
  值得注意：一条消息通过RoutingKey与同一条队列多次匹配时，topic交换机和direct交换机都只会向该队列发送一次该消息。
